# 第3章：Hprof之心

吕修回到APM守望塔后，打开了D-07区那座建筑的内存快照。完整的Hprof文件——在建筑白屏崩溃时由APM系统自动捕获并上传。

他用Hprof分析工具打开了文件。对象树展开，引用链从GC Root开始一级级向下追——一个静态的HashMap→持有Activity引用→Activity持有DecorView→DecorView持有整个View树。View树中占内存最大的几项：三颗被重复加载的Bitmap，每颗大约占用几兆内存；一组因未正确释放而被缓存下来的页面骨架请求区块。

他截了图，标注了泄漏链路径。

但他停下来想了一会儿——每次都是崩溃后手动分析快照发现问题。三周前D-09区，两周前C-11区，上周D-07区，今天同一座楼。如果每次都是人工追，时间一长，要么追踪不到位，要么干脆遗漏。他在塔内的工作位上打开了开发终端，开始写一个自动分析内存快照的模块。

规则很简单：在每次用户关闭Activity后，触发一次内存快照，与上一个快照对比。如果发现有新增的Activity实例存活超过一定时间——标注为"待确认泄漏"。

他把这个模块作为APM守望塔的一个扩展功能提交到了应用商店自动更新。扩展程序在塔的核心机柜中完成部署，第一次对全城所有建筑的Activity生命周期进行了遍历扫描。

扫描结果在几小时后以列表形式返回了名单：外环有若干座建筑存在死后持续存活的Activity实例。

吕修从塔顶俯瞰着外环低矮的建筑轮廓。

"如果模块能在每次用户关闭Activity后自动捕获一次快照，并在快照中自动检测泄漏的Activity实例——我就能在用户还没有感觉到卡顿之前就知道泄漏在哪里。"他对着自己的终端补完了这句没有写出注释的备注。
