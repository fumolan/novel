# 第2章：判空陷阱

三天后，同一栋楼的白屏报告在APM守望塔的数据面板上亮了，时间比上一次提前了四小时。

吕修到现场时，大楼安静得不正常。大屏入口模块垂着头蹲在楼门口。

"判空加上了，"他说，"但更严重了。加了判空之后三个小时，楼里一半以上的模块同时掉线——不是白屏，是直接退出。"

吕修没有急着打开堆栈。他先翻了版本变更日志——"修复D-07-011楼偶现NullPointerException，在onResume入口处增加判空处理。"

他找到了被修改的那段代码。

```
if (xxData != null) {
    xxData.loadToView();
}
```

判空保护了loadToView()不被空指针打断。但如果xxData为空，if块内的代码不执行——视图没有加载——用户看到白屏。块结束后调用了下一段逻辑——那段逻辑假设xxData已经加载完毕，需要用它做数据源。数据源为空——下游方法使用了一个未经有效性检测的空对象——直接闪退。

"判空不治本，"吕修说，"而且后面的逻辑在空值情况下依然会执行。你加了一个null检查阻止了第一行代码出错，但你让整个数据流的后续逻辑在完全没有数据的状态下运行。"

"那不加判空能怎么办？"

"不是不加判空——是不要在没有理解数据从哪来之前就先加一个if。null检查没有问题。问题是null检查不应该替代对"为什么数据会有空值"的根因分析。"

他打开了建筑的内存快照，回溯了那个static集合持有的Activity对象。Activity实例的主键特征显示，它在退出前曾经通过一个异步回调向服务器请求最新数据——回调在Activity销毁后才返回。Activity销毁后它的引用因为没有从静态集合中被移除，所以它继续存活——回调使用静态集合中的这个实例更新了UI——更新UI时发现数据字段为空——赋值失败——触发空指针。

修复方案：不是加判空。是在Activity的onDestroy方法中从静态集合中移除自身引用。Activity正常回收——回调不再更新已销毁的Activity——问题消失。十七行代码变成一行。

吕修提交了工单。APM守望塔的数据库记录了这条修复路径。

他离开时，大楼入口模块站在门口，看着他的背影说："你早来三天就好了。"

吕修没有回头。
